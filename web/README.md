# Logos Web

A browser client for Logos that runs the **real** protocol crates compiled to
WebAssembly — no crypto is reimplemented in JavaScript, so the web client is
byte-compatible with the iOS app by construction. Private keys are generated in
the browser and never leave the device.

**EXPERIMENTAL — UNAUDITED.** See the repository [README](../README.md) and
[`docs/PROTOCOL.md`](../docs/PROTOCOL.md).

## How it fits together

This mirrors the iOS structure:

| | iOS | Web |
|---|---|---|
| bindings crate | [`crates/logos-ffi`](../crates/logos-ffi) (UniFFI) | [`crates/logos-wasm`](../crates/logos-wasm) (wasm-bindgen) |
| app | [`ios/`](../ios) | `web/` (this dir) |

The `logos-wasm` crate exposes the engine to the browser the same way `logos-ffi`
exposes it to Swift. JavaScript owns transport and persistence — exactly as Swift
does on iOS.

- **`public/`** — the static client. `index.html`, `styles.css`, `app.js`. The
  WASM module + bindings (`public/wasm/`) and brand images are generated by
  `build-wasm.sh` and are gitignored.
- **`worker.js`** — Cloudflare Worker. Serves the static client and proxies
  `/v1/*` to the relay **same-origin**, so the browser needs no CORS and the
  relay needs no changes. It also sets a strict CSP and security headers — the
  primary control against XSS, which matters because keys live in the browser.
- **`wrangler.toml`** — deploys the Worker + assets to a custom domain.

The relay is a dumb store-and-forward of sealed blobs; all E2EE happens in the
browser, so neither the Worker nor the relay ever sees plaintext.

## Build

```bash
./build-wasm.sh
```

Builds `crates/logos-wasm` for `wasm32-unknown-unknown` (pinning the
`getrandom_backend="wasm_js"` cfg that ml-kem's RNG path requires), runs
`wasm-bindgen --target web`, and stages brand assets into `public/`. Requires the
repo's pinned Rust toolchain; the script fetches a matching `wasm-bindgen` CLI if
one isn't present.

## Deploy

```bash
# Provide a Cloudflare API token with edit rights to the zone, then:
CLOUDFLARE_API_TOKEN=… npx wrangler deploy
```

## Status

Live at **https://logos.manticthink.com**.

**Phase 1 (done):** in-browser identity creation → registration to the relay →
recovery seed; identity persisted locally (secret material AES-GCM-encrypted at
rest); peer lookup + safety-number comparison.

**Phase 2 (done):** 1:1 end-to-end-encrypted messaging — the PQXDH handshake,
Double Ratchet, and sealed-sender send/receive over the relay, with
per-conversation session state. The stateful `WasmClient` (in `crates/logos-wasm`)
owns the crypto and ratchet sessions; JS owns transport and persistence. Send and
receive are split into pure-compute steps (`prepare_send` / `process_incoming`)
with JS doing the async relay `fetch` between them, so the engine stays
synchronous and wasm-friendly. Every wire type comes from `logos-proto`, so the
bytes are identical to iOS by construction. Verified three ways: a native Rust
round-trip test, a Node integration test against the live relay, and a headless
Chromium UI test (two users, under the production CSP).

**Phase 3 (next):** groups (sender-key), attachments, the AI layer — plus the
remaining hardening (passphrase-derived at-rest key, restore-from-seed, BIP39
24-word phrase, one-time-prekey replenishment). See `HANDOFF.md`.

## A note on "private" on the web

Browser storage can't match iOS's hardware-backed Keychain. At-rest encryption
protects stored keys from offline scraping, but a live cross-site-scripting bug
in the origin could still reach them — which is why the CSP is strict. Treat the
web client accordingly until it has been reviewed and exercised in real browsers.
